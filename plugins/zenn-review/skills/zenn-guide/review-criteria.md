# レビュー基準・ベストプラクティス

## レビューの4観点

### 1. 具体性（Specificity）

記事の内容が十分に具体的で、読者が実際に活用できるかを評価する。

**チェックポイント:**
- コード例が含まれているか（tech記事の場合）
- 手順が具体的に記述されているか
- 抽象的な説明だけで終わっていないか
- 入出力の例が示されているか
- バージョン情報が明記されているか

**評価基準:**
| スコア | 説明 |
|--------|------|
| 5 | コード例・手順・入出力例が豊富で、そのまま再現可能 |
| 4 | 主要な部分にコード例があり、手順も明確 |
| 3 | コード例はあるが一部不足、または手順が曖昧な箇所がある |
| 2 | コード例が少なく、抽象的な説明が多い |
| 1 | ほぼ抽象的な説明のみで、実践に移しにくい |

### 2. 正確性（Accuracy）

技術的な内容が正確であるかを評価する。

**チェックポイント:**
- コード例が構文的に正しいか
- API の使い方が正しいか
- 用語の使い方が正確か
- 非推奨（deprecated）な方法を推奨していないか
- バージョン固有の情報が正しいか

**評価基準:**
| スコア | 説明 |
|--------|------|
| 5 | 技術的に正確で、コード例も全て正しい |
| 4 | ほぼ正確だが、軽微な不正確さがある |
| 3 | 主要な部分は正確だが、一部に誤りがある |
| 2 | 複数の技術的な誤りがある |
| 1 | 根本的な誤りがある、または誤解を招く内容 |

### 3. 完全性（Completeness）

テーマに対して必要な情報が網羅されているかを評価する。

**チェックポイント:**
- 前提条件・環境情報が記載されているか
- エラーハンドリングや注意点が言及されているか
- 参考リンクが適切に提示されているか
- 読者が疑問に思いそうな点がカバーされているか
- 結論やまとめがあるか

**評価基準:**
| スコア | 説明 |
|--------|------|
| 5 | テーマに必要な情報が全て網羅されている |
| 4 | 主要な情報は網羅、軽微な不足がある |
| 3 | 基本的な情報はあるが、いくつかの重要な情報が不足 |
| 2 | 重要な情報が複数欠けている |
| 1 | 必要最低限の情報すら不足している |

### 4. 一貫性（Consistency）

文体・用語・構成が記事全体で一貫しているかを評価する。

**チェックポイント:**
- 文体が統一されているか（です・ます調 / だ・である調）
- 技術用語の表記が統一されているか
- 見出しレベルの使い方が適切か
- コードブロックのスタイルが統一されているか
- 語調やトーンが一貫しているか

**評価基準:**
| スコア | 説明 |
|--------|------|
| 5 | 完全に統一された文体・用語・構成 |
| 4 | ほぼ統一されているが、軽微なブレがある |
| 3 | 概ね統一されているが、数箇所で不統一 |
| 2 | 文体や用語にかなりのブレがある |
| 1 | 統一性がほとんどない |

## 日本語技術記事の表記ルール

### 文体

- **です・ます調** と **だ・である調** を混在させない
- 記事全体で統一すること

### 技術用語の表記

よく揺れる用語の推奨表記:

| 推奨 | 非推奨例 |
|------|----------|
| JavaScript | Javascript, javascript, JS（文中） |
| TypeScript | Typescript, typescript |
| React | react（文中） |
| Node.js | NodeJS, Nodejs, node.js |
| GitHub | Github, github |
| VS Code | VSCode, vscode |
| API | api（文中） |
| URL | url（文中） |
| OS | os（文中） |
| CLI | cli（文中） |
| npm | NPM |
| Docker | docker（文中） |
| Kubernetes | kubernetes, k8s（正式名として） |
| macOS | MacOS, macos |
| Linux | linux（文中） |
| Windows | windows（文中） |

### 日本語表記

- 全角・半角の使い分けを統一する
- 日本語と英数字の間にスペースを入れる（推奨）
- 句読点は「、」「。」で統一（「，」「．」と混在させない）
- カタカナ語の長音記号を統一する（例: 「サーバー」vs「サーバ」）

### 見出し構成

- `#`（h1）は使用しない（タイトルがh1扱い）
- `##`（h2）から始める
- 見出しレベルを飛ばさない（h2 → h4 は不可、h2 → h3 → h4）
- 見出しの粒度を揃える

## レビューレポートのフォーマット

```markdown
## レビューレポート: {ファイル名}

### スコアサマリー

| 観点 | スコア | 概要 |
|------|--------|------|
| 具体性 | X/5 | ... |
| 正確性 | X/5 | ... |
| 完全性 | X/5 | ... |
| 一貫性 | X/5 | ... |
| **総合** | **X/5** | ... |

### 指摘事項

#### ERROR（修正必須）
- ...

#### WARNING（修正推奨）
- ...

#### INFO（改善提案）
- ...

### 改善案

1. ...
2. ...
3. ...
```
